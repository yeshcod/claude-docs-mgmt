# Product Requirements Documents (PRDs)

Archived requirements docs from `/develop` sessions or feature planning. Append-only — once a PRD is shipped, it stays as a historical record.

## Naming

`YYYY-MM-DD-slug.md` — date of finalization (not start), 3-5 word slug, lowercase.

## When to write a PRD

- A `/develop` session produced one in Phase 1 — save it here.
- A feature is non-trivial (>1 week of work, >3 files, requires UX + backend + tests) and needs alignment before build.
- Multiple stakeholders need to agree on scope and non-goals.

Do NOT write a PRD for:
- Bug fixes (use BUGS.md entry).
- Small config tweaks.
- Refactors (use a brief note or ADR if architectural).
